2019.07月总结

发表于 2019-08-05 | 更新于 2019-08-08 | 分类于 月总结

这是打算写月总结的第一篇文章，真实日期为2019.07.16~2019.07.31，本文的目的在于从多个不同的纬度在单月时间内来复盘、沉淀自已

架构能力

MVVM

架构分层核心：数据管理者，数据加工者，数据展示者，后面的演化只是针对某一层的细化分解
在没有使用MVVM前，即在MVC中我们也会采用一些手段来降低Controller的复杂度，比如从Controller中提取网络请求先关，提取dataSource、delegate等手段，其实我们在“拆” 的过程中就是在向着各种MV“X”框架转化的过程。

ViewModel与model

在日常的开发过程中，其实就是从抽象到形象 不断相互转化的过程，我们把JSON数据提取出来并对应到View上展示，这就是一个从抽象到形象的过程。反之亦然，ViewModel就是Model层在获得数据之后，通过C，再通过VM最终展示到到View层的中间产物。反过来就是View产生的用户行为，通过VM 到C，再到Model。这里例举两个常见的场景

到这里就会遇见一个问题：按照上面的例子来看，正常情况下应该每一个Model都对应一个ViewModel了。

事实也确实应当如此，但是如果这样做就会有大量的转换工作要做比如，网络请求结束，我们拿到一个list，我们要经过List字典对象转化为Listmodel对象，再到List ViewModel对象。

这显然不是我们想要的，所以在绝大部分情况下，我们是跳过了后面的第二步，只有在一些复杂需求的页面会采用这样方式：

比如朋友圈页面，微博时间轴页面等，这些内容元需要经过大量的计算，对象组合，甚至缓存策略。此时正是ViewModel大显身手的时候，所以面对这种场景还是乖乖的把每一个Model都对应上ViewModel吧。
这里重新总结下ViewModel的使用方法：

  • 简单情况下，我们可以一个Controller的View对应个ViewModel。此时ViewModel要负责vc.view和subviews的数据形象过程。
  • 网络请求也可放在ViewModel中，如果要写在ViewModel,那么对外回调做好是block(并且是处理好的比如一些异常的提示等)，这样对于单元测试是有利，也可提取单独的data store 层
  • 在复杂的场景下（列表内容多元化，复杂化）,这样场景是要把每一个Model层都包一层ViewModel的。
